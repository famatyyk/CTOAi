# VPS Agent Outputs Runbook

**Last Updated:** 2026-05-21T15:01:40+00:00

## Purpose
Quick map of where to find latest artifacts generated by CTOA agents on VPS.
Use this after a run when you need proof of what was produced, by whom, and when.

## Primary Locations

### 1) Generated output files
- Main directory: `/opt/ctoa/generated`
- Typical Intel path: `/opt/ctoa/generated/intel_target`
- Expected content: generated `.lua` modules and related output files.

### 2) Agent and runner logs
- Orchestrator log: `/opt/ctoa/logs/agents-orchestrator.log`
- Runner log: `/opt/ctoa/logs/runner.log`
- Health log: `/opt/ctoa/logs/health-live.log`
- Intel watcher log: `/opt/ctoa/logs/intel-news-watcher.log`
- Intel API log: `/opt/ctoa/logs/intel-news-api.log`

### 3) Database evidence
- `agent_runs` table: who ran, status, finish time, message
- `modules` table: generated modules, status, quality
- `daily_stats` table: daily quality and volume summary
- `servers` table: source URLs and current pipeline status

### 4) Training reports
- Directory: `/opt/ctoa/runtime/training-reports`
- Files:
  - `latest.md`
  - `latest.json`
  - historical `auto-trainer-YYYYMMDD-HHMMSS.md`

## 60-Second Check (copy/paste)

```bash
cd /opt/ctoa

echo "=== latest generated files ==="
find generated -type f -printf "%TY-%Tm-%Td %TH:%TM %p\n" | sort -r | head -40

echo "=== orchestrator log tail ==="
tail -n 120 /opt/ctoa/logs/agents-orchestrator.log

echo "=== runner log tail ==="
tail -n 120 /opt/ctoa/logs/runner.log

echo "=== latest agent runs (db) ==="
psql "postgresql://ctoa:${DB_PASSWORD}@127.0.0.1:5432/ctoa" \
  -c "SELECT agent, status, finished_at FROM agent_runs ORDER BY id DESC LIMIT 20;"
```

## Deep Check (DB)

```bash
# Latest generated module rows
psql "postgresql://ctoa:${DB_PASSWORD}@127.0.0.1:5432/ctoa" \
  -c "SELECT task_id, output_file, quality_score, status, validated_at FROM modules ORDER BY validated_at DESC NULLS LAST LIMIT 30;"

# Daily quality snapshot
psql "postgresql://ctoa:${DB_PASSWORD}@127.0.0.1:5432/ctoa" \
  -c "SELECT dt, modules_generated, programs_generated, avg_quality, launcher_day FROM daily_stats ORDER BY dt DESC LIMIT 7;"

# Server intake state
psql "postgresql://ctoa:${DB_PASSWORD}@127.0.0.1:5432/ctoa" \
  -c "SELECT id, url, status, updated_at FROM servers ORDER BY id DESC LIMIT 20;"
```

## VPS Rollout Checklist (ctoa-intel-news-*)

Use this after deploying new unit files or renaming services from old scan naming.

1. Sync unit files to `/etc/systemd/system/`:

```bash
sudo cp /opt/ctoa/deploy/vps/systemd/ctoa-intel-news-api.service /etc/systemd/system/
sudo cp /opt/ctoa/deploy/vps/systemd/ctoa-intel-news-watcher.service /etc/systemd/system/
sudo cp /opt/ctoa/deploy/vps/systemd/ctoa-intel-news-watcher.timer /etc/systemd/system/
```

2. Reload systemd and reset failed state:

```bash
sudo systemctl daemon-reload
sudo systemctl reset-failed
```

3. Enable and start intel units:

```bash
sudo systemctl enable --now ctoa-intel-news-api.service
sudo systemctl enable --now ctoa-intel-news-watcher.timer
```

4. Verify active state:

```bash
systemctl is-active ctoa-intel-news-api.service
systemctl is-active ctoa-intel-news-watcher.timer
systemctl list-timers --all | grep ctoa-intel-news-watcher
```

5. Validate API and logs:

```bash
curl -fsS http://127.0.0.1:8890/health
journalctl -u ctoa-intel-news-api.service -n 80 --no-pager
journalctl -u ctoa-intel-news-watcher.service -n 80 --no-pager
```

6. Optional one-command rollout from workstation:

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops/ctoa-vps.ps1 -Action Setup24x7
powershell -ExecutionPolicy Bypass -File scripts/ops/ctoa-vps.ps1 -Action ShowSystemHealth
powershell -ExecutionPolicy Bypass -File scripts/ops/ctoa-vps.ps1 -Action ShowServiceRestarts
```

7. Mobile console verification:

```bash
curl -H "X-CTOA-Token: ${CTOA_MOBILE_TOKEN}" http://127.0.0.1:8787/api/status
curl -H "X-CTOA-Token: ${CTOA_MOBILE_TOKEN}" http://127.0.0.1:8787/api/agents/intel/report
```

Expected: `ctoa-intel-news-api.service` is `active`, watcher timer is active, `/health` returns OK, and mobile status reports intel API health.

## Mobile Console Paths (remote quick view)

If mobile console is running, use token-authenticated endpoints:

- `GET /api/agents/intel/report` -> recent runs, quality, failed modules, trainer actions
- `GET /api/dashboard` -> servers/modules/stats/top snapshot
- `GET /api/agents/auto-trainer/latest` -> latest training reports
- `POST /api/agents/intel/run` -> triggers orchestrator and returns generated files list
- `GET /api/logs?target=agent_orchestrator&lines=120` -> orchestrator tail

Example:

```bash
curl -H "X-CTOA-Token: ${CTOA_MOBILE_TOKEN}" http://127.0.0.1:8787/api/agents/intel/report
curl -H "X-CTOA-Token: ${CTOA_MOBILE_TOKEN}" http://127.0.0.1:8787/api/dashboard
curl -H "X-CTOA-Token: ${CTOA_MOBILE_TOKEN}" "http://127.0.0.1:8787/api/logs?target=agent_orchestrator&lines=120"
```

## If You See No New Outputs

1. Verify services are active (commands above).
2. Trigger one run manually:

```bash
systemctl start --no-block ctoa-agents-orchestrator.service
sleep 3
tail -n 120 /opt/ctoa/logs/agents-orchestrator.log
```

3. Check DB for failed statuses in `agent_runs` and `modules`.
4. Check disk and permissions:

```bash
df -h /
ls -ld /opt/ctoa/generated /opt/ctoa/logs
```
